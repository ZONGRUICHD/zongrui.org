# ZongRui Articles edge layer

The Pages project remains the public origin. `functions/_worker.js` provides three narrowly scoped behaviors:

- `/api/articles/*` proxies to the private self-hosted Articles service.
- `/articles/*` serves the SPA shell and injects article-specific metadata and a server-readable article body.
- `/sitemap.xml` is generated by the Articles service, with a minimal static fallback.

## Required Pages environment variables

Configure both Preview and Production in **Workers & Pages → zongrui-org → Settings → Variables and Secrets**:

| Variable | Type | Value |
| --- | --- | --- |
| `ARTICLES_ORIGIN_URL` | plain variable | `https://articles-origin.zongrui.org` |
| `ARTICLES_ORIGIN_SHARED_SECRET` | encrypted secret | Optional origin token when Access service tokens are unavailable |
| `CF_ACCESS_CLIENT_ID` | encrypted secret | Cloudflare Access service-token client ID |
| `CF_ACCESS_CLIENT_SECRET` | encrypted secret | Cloudflare Access service-token client secret |
| `VITE_TURNSTILE_SITE_KEY` | build variable | Public Turnstile widget site key |

The preferred production setup is a Cloudflare Access **Service Auth** application where only the Pages service token is allowed. If Access cannot yet be enabled, set `ARTICLES_ORIGIN_SHARED_SECRET` in both Pages and the origin service; the proxy sends it as `X-ZR-Origin-Token`. Either credential mode may be enabled, and both can coexist during migration. The Worker strips visitor-supplied Access, origin-token, forwarding and visitor-IP headers before adding its own trusted values.

`VITE_TURNSTILE_SITE_KEY` is public but must be present in the Pages **build** environment because Vite embeds it into the client bundle. The Turnstile secret belongs only in the backend environment.

For local development, copy `.dev.vars.example` to `.dev.vars` and point `ARTICLES_ORIGIN_URL` at the loopback backend. All origin credential fields may remain empty only for localhost origins. Set `VITE_TURNSTILE_SITE_KEY` in a local uncommitted `.env.local` when testing comments.

## Local verification

```bash
npm run build
npx wrangler pages dev dist
```

Test direct loads at `/articles`, `/articles/example-slug`, `/articles/console`, `/api/articles/v1/articles`, and `/sitemap.xml`.

Public article detail JSON is cached at the edge for five minutes. The main
list, tags, RSS and sitemap use a one-minute fresh lifetime so scheduled posts
become visible promptly; searched, filtered and cursor-based lists bypass the
Cache API so targeted invalidation cannot leave query variants behind. A
seven-day stale copy is used only for rate-limit or server failures when the
home server is unavailable. Comments, authentication and Console traffic are
never cached. Mutating backend responses may send `X-ZR-Cache-Invalidate` with
comma-separated public paths for best-effort targeted eviction in the current
edge location; other detail copies naturally expire within five minutes.

The implementation uses Pages Functions file-system routing. `functions/[[path]].js` is the deployed catch-all adapter; `functions/_worker.js` contains the Worker-style implementation. `_routes.json` limits invocations to Articles API/page routes and the dynamic sitemap, leaving ordinary static assets on Pages' asset path.
